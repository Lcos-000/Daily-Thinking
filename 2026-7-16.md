# Continuous Batching 下的显存约束与请求抢占

> **核心挑战**：在 Continuous Batching 机制下，如何在高吞吐与低风险间取得平衡？当 GPU 显存成为瓶颈，调度器应如何在“接纳新请求”与“保护旧请求”之间做出智能决策？

---

## 1. 问题原点：从静态预留到动态增长的范式转移

面对“显存有限，KV Cache 持续增长”的难题，第一反应往往是进行**准入控制**——预估新请求的内存需求，决定是否接纳。

但“预判”本身就是一个陷阱。用户的 `max_tokens` 只是上限，实际生成长度往往远小于此。这导致两种天然的、但都有缺陷的策略：

- **按最坏情况（max_tokens）预留**：看似安全，但代价高昂。在生产环境中，多数请求的实际生成长度远低于上限，这会严重低估 Batch 能容纳的请求数，导致大量显存为“用不到的部分”闲置，**直接违背 Continuous Batching 最大化 GPU 利用率的初衷**。
- **按平均情况估算**：能提升并发量，但没有任何兜底，一旦多个请求同时超出平均值，就会触发显存溢出（OOM），导致服务崩溃。

**关键洞察**：真正的解决方案，不是设计一个更精妙的预估算法，而是**彻底改变资源分配的模式**。

既然显存占用是随生成过程逐步增长的，为什么不把分配方式从“请求准入时一次性预留未来全部所需”，改为“生成一个 token 才分配一块显存”？这与操作系统从固定分区进化到**虚拟内存与按需分页**的思路异曲同工。在 LLM 推理引擎中，这一思想的具体实现便是 **PagedAttention**（vLLM 的核心创新），它将 KV Cache 显存划分为固定大小的 Block，按需分配，用完即回收。

由此，风险从“准入阶段”转移到了“运行阶段”。我们不再需要悲观的预估，但必须依赖两种能力来承接风险：**迅捷高效的抢占**和**有效的进度保留**。

---

## 2. 核心权衡：被抢占的进度，保留还是丢弃？

当一个正在生成到一半的请求因显存压力被抢占，它的处理方式成为一个核心决策点。一个直接的类比是训练任务中的 Checkpoint 机制，但两者的时间尺度和代价特性截然不同：

- **训练 Checkpoint**：保存到持久化存储，恢复可能需要等待数小时甚至更久。操作本身的开销（写入磁盘）相对于训练总时长可以接受。
- **推理抢占**：是秒级甚至亚秒级的决策，恢复通常在几百毫秒内完成。将 KV Cache 写入磁盘再读回的开销，可能比直接丢弃、恢复时重新计算还要慢。

这迫使我们必须建立一个精确的成本模型，来指导决策。候选的“进度保留”方案有三种，构成了一个完整的分层存储与回退体系：

1.  **方案 A：重量级保存 (Swap to CPU Memory)**
    - **操作**：将 KV Cache 从 GPU 显存换出到 CPU 内存，恢复时再换入。
    - **成本**：`T_换入换出(n) = KV_Cache_Size(n) / PCIe_Bandwidth × 2`（写+读两次搬运）。

2.  **方案 B：轻量级丢弃 (Evict & Recompute)**
    - **操作**：直接释放 GPU 显存，恢复时对之前生成的 n 个 token 重新执行一次 Prefill。
    - **成本**：`T_重新计算(n) = Time_Prefill(n)`
    - **反直觉点**：由于 Prefill 阶段对输入 token 进行高度并行化处理，其速度极快。对于小 n，重新计算可能比通过慢速 PCIe 总线搬运数据更快。

**量化决策临界点**：一个请求被抢占后，选择方案 A 还是方案 B，取决于已生成的 token 数量 `n`。存在一个临界点 `N`：

- **当 `n < N` 时**：`T_重新计算(n) < T_换入换出(n)`。直接丢弃，快速重算更划算。
- **当 `n > N` 时**：`T_重新计算(n) > T_换入换出(n)`。此时生成 token 的沉没成本足够高，值得承受 PCIe 带宽的传输开销，将其换出到 CPU 内存保护起来。

这个临界点 `N` 取决于具体的硬件（PCIe 带宽、GPU 算力）和模型大小，需要实测校准。

---

## 3. 多维权重修正：超越单纯的成本计算

上述成本模型提供了一个客观的“经济”判断基准。但真实的调度系统还需要考虑人类赋予的主观价值。

此时，**请求优先级**作为权重叠加到决策上，它**调整了临界点本身的位置**，而不是推翻“存不存”这个判断依据。

- 对于**高优先级**请求，我们可能愿意在它还未到达临界点 `N` 时，就额外花费 PCIe 带宽把它换出保存，以换取它被恢复时更低的延迟（直接续生成，无需 Prefill 重算）。
- 对于**低优先级**请求，即使它刚刚超过临界点，我们可能也选择直接丢弃，以节省有限的 CPU 内存和 PCIe 带宽资源给更重要的请求。

这就像一个“头等舱优先登机/优先行李”机制，有明确的成本底线，但价值导向的干预可以改变资源的分配顺序。

---

## 方法论沉淀

1.  **反“预留”思维，拥抱按需分配**：面对资源约束和预测的不确定性，“按最坏情况预留”是牺牲效率换取安全的懒政。更优解往往是将分配从“准入阶段的静态预估”转为“运行时的动态增长与回收”，并用高效的运行时机制（抢占/恢复）来承接风险。这是 `PagedAttention` 等先进技术背后的深层哲学。

2.  **慎用类比，量化成本**：“Checkpoint”这类跨场景的概念不能直接套用。必须根据具体场景的时间尺度（训练的小时级 vs 推理的秒级）和操作特性（反直觉的快速 Prefill 重算），建立量化的成本模型，寻找决策临界点。同一个“保存-恢复”目标，在不同的计算特性下，最优策略可能截然相反。
