```markdown
# Day 8 思考链路记录：分布式调度中的声明式、幂等与状态收敛

> **原题**：调度器采用主备架构，所有状态存于外部共享存储。Leader 正在处理某个调度请求，已在内存中算出分配结果（节点 A），但还未写入共享存储时，Leader 所在机器突然崩溃，新 Leader 接管。新 Leader 对旧 Leader 处理到哪一步毫无感知，这个任务现在处于什么状态？如果调用方用幂等 key 重试，会不会出现"任务从未被真正分配、但也永远查不到失败原因"的悬空状态？该如何设计，让"处理到一半、进程突然消失"不会导致任务永久卡在不明状态？

---

## 一、讨论起点：是否可用 K8s 模式回避问题？

**第一反应**：
直接采用 K8s 的声明式调度就不会出现这个问题，这个场景恐怕不存在。

**纠正**：
这个回应是用"更换系统设计"来回避问题，而不是解决问题——换了场景问题依然存在，只是转移了位置。

深入分析 K8s 的声明式调度为什么能规避这个问题：
- 调用方提交请求时，"期望状态"本身就被先持久化写入 etcd
- 调用方拿到的响应只代表"期望状态已被记录"，不代表调度已完成
- kube-scheduler 是异步的、持续 watch etcd 变化的控制循环
- 若处理到一半崩溃，因为"期望状态"完好地留存在 etcd 里，新实例起来后重新发现待调度对象，直接重新处理即可，没有任何东西丢失

这个设计能生效的前提，正是**"调度决策"与"调用方的同步请求-响应"被彻底解耦**，期望状态本身是一份可反复读取、反复重试的持久化记录。

**追问**：你自己设计的调度器，请求处理的第一步是否也应该先持久化一份类似"期望状态"的记录，而不是"来了就在内存里处理，处理完才落盘"？

## 二、方案对比：先持久化意图 vs 边处理边存状态

**回答**：
应当先把请求内容存下来，彻底完成调度之后再删掉，或者采用 K8s 的声明式思想。

**认可 + 追问**：
方向正确，抽象出了"先持久化意图，再异步处理，处理完再清理/标记完成"这一本质。

给出两种时序方案对比：
- **方案 A**：先存请求为"待处理" → 返回"已接收"响应 → 异步处理 → 处理完标记"已完成"
- **方案 B**：一边处理一边存中间状态

**追问**：哪种方案能真正保证进程随时崩溃都不丢状态？如果崩溃恰好发生在"存请求"这个动作执行到一半（如正在写共享存储时网络中断），这份"存下来的请求内容"本身会不会也变成一份不完整、状态不明的记录？"存请求"这个动作本身，是否也需要是原子的、要么完全成功要么完全没发生？

## 三、入口处的幂等保障

**回答**：
方案 A 更好，因为这个动作时效性比较强，做错了就重来。

**纠正**：
"时效性"这个理由与问题不对应——问题的核心不是速度，而是**"存请求"这个动作本身是否也存在半途崩溃导致状态不完整的风险**。

具体推演：
- "存入共享存储"的写操作本身可能因网络抖动等原因失败或中断
- 调用方可能收到错误或超时进而重试
- 这其实是 Day 4 讨论过的原始问题的翻版——"存请求"这个动作，同样需要用幂等机制来保证：调用方带幂等 key，调度器据此判断是否已经存过，避免重复写入

**阶段结论**：整个调度流程要做到崩溃后能安全恢复，靠的不是某一环节做得多快多好，而是**每一个可能崩溃的环节（接收请求、挑节点、写分配结果）都要能用同一套幂等/原子写入的思路兜底**——每一步都需要具备"重新执行一遍，结果依然正确"这一性质。

**追问**：为什么"声明式 + 每一步都幂等"能让"进程随时可能崩溃"这件事变得不再可怕，能否用一句话总结？

## 四、声明式与幂等的配套关系

**回答**：
声明式保证了这个请求不丢，任务最终一定会完成；幂等保证了不会重复。

**补充连接点**：
这两句话本身都对，但它们是同一件事的两面，缺一不可，而非两个独立的保证。

**完整逻辑链**：

1. **声明式（先持久化意图）** 的作用：
   - 保证"任务的存在本身"不会因进程崩溃而丢失
   - 总会有人（新 Leader 或恢复后的自己）重新捡起处理

2. **为什么需要幂等**：
   - "重新捡起处理"这个动作可能被执行不止一次
   - 因为无法精确知道"上次处理到哪一步"
   - 最安全的做法是"当作没处理过，重新走一遍"
   - 这就可能导致重复分配、重复扣资源

3. **幂等在此时发挥的作用**：
   - 保证的不是"请求不丢"
   - 而是"同一个任务被处理 N 次，产生的效果和处理 1 次一样"
   - 从而抵消声明式模式必然带来的"可能被重复处理"这一副作用

**精确总结**：
- **声明式**解决"任务会不会丢"
- **幂等**解决声明式模式必然带来的副作用——"任务会不会被重复处理、处理多次会不会出问题"

**两者是配套关系**：
- 只用声明式不配幂等 → 会导致重复调度
- 只用幂等不配声明式 → 进程崩溃时中间状态依然会丢，任务可能永远无人处理

## 五、K8s 的幂等实现：基于状态收敛的幂等

**追加提问**：
K8s 做了幂等吗？

**展开回答**：
K8s 从架构设计上天然要求幂等，但其实现方式与"幂等 key + 状态表查重"不同，而是依靠**基于状态对比的协调循环（Reconciliation Loop）**。

**核心机制**：
- 每个 controller 的核心逻辑是不断对比"实际状态（Status）"与"期望状态（Spec）"之间的差异
- 有差异就采取行动缩小差异
- 无差异则不做任何事

**为什么天然幂等**：
- 不是因为"记住了处理过"
- 而是因为"每次都重新对比一遍现状"
- 例如：某次写入 `nodeName` 的动作被重复执行两次，第二次执行时会先检查该字段是否已被写入，若已写入则直接跳过

**与"幂等 key + 查重表"的区别**：
- K8s 不需要专门的"是否处理过"记录表
- "实际状态"本身就充当了这张记录表
- 判断是否需要处理，靠的是直接检查"目标状态是否已经达成"，而非比对一个幂等标识

**这种模式被称为**：**"基于状态收敛的幂等"（State Convergence / Level-Triggered）**

**追问**：调度器场景中，"幂等 key 查重表"和"基于状态收敛的判断"哪种更适合？是否可以结合？任务的分配记录本身是否可以充当"当前状态"，无需额外维护幂等表？

## 六、两种模式的定位与分工

**回答**：
状态收敛显然更优美。对于业务逻辑不强的算力调度而言，状态收敛更好，因为它的目的是保护一个状态，而不是需要一个具体的任务结果。

**平衡视角的补充**：
这一提炼准确，点出了两种模式背后不同的建模视角：
- **以事件/请求为中心**：是否处理过这个请求（幂等 key 模式）
- **以状态为中心**：这个对象现在是什么样子（状态收敛模式）

**但状态收敛模式的代价**：
- 判断"是否已达成目标状态"，前提是能查到该任务对应的完整状态记录
- 如果请求刚到、调度器还未来得及把任务写入共享存储就崩溃——此时根本没有"状态"可供对比，无法判断"达成没达成"
- 因此状态收敛模式依然依赖一个前提：**必须先有一份持久化的"期望状态"记录存在**（对应 K8s 中 Pod 声明写入 etcd 这一步）
- 而这一步的持久化写入本身，仍然要处理"写入过程中崩溃、调用方是否要重试"的问题——这依然是幂等写入要解决的场景

**最终结论**：
- **状态收敛**解决的是"处理逻辑该如何设计成可重复执行"——无需记录是否处理过，只需对比当前状态与目标状态的差距
- 但解决不了"期望状态本身第一次被创建"这一步的幂等问题——这一步依然需要幂等 key / 原子写入机制兜底

**两者不是替代关系，而是分工在流程的不同阶段**：
- **入口处（创建期望状态）** → 靠幂等 key / 原子写入
- **内部处理循环（推进现状趋近目标）** → 靠状态收敛

## 七、方法论沉淀

设计高可用系统时，"进程随时可能崩溃"这一现实约束要求架构在**两个层面**都具备恢复能力：

### 入口层面
- **问题**：如何保证一个请求被安全地转化为一份持久化的意图，不因写入过程中的崩溃而产生重复或丢失
- **手段**：依赖幂等机制（幂等 key、原子写入、CAS 等）

### 处理层面
- **问题**：如何保证处理逻辑可以被任意次重新执行而不产生副作用
- **手段**：依赖状态收敛或幂等写入

### 核心原则
两者服务于流程的不同阶段，**缺一不可**。简单地用其中一种替代另一种，都会在某个环节留下"进程崩溃导致状态不一致"的隐患。

## 八、完整逻辑链路回顾

```
调用方带幂等 key 提交请求
        │
        ▼
┌─────────────────────────┐
│  入口层：幂等写入        │
│  - 原子写入期望状态记录  │
│  - 幂等 key 防止重复创建 │
│  - 返回"已接收"响应     │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  处理层：状态收敛        │
│  - 读取期望状态         │
│  - 对比实际状态         │
│  - 有差异则推进         │
│  - 无差异则跳过         │
│  - 任意次重放均安全     │
└─────────────────────────┘
```

**一句总结**：声明式解决"任务会不会丢"，幂等解决"重试会不会乱"，两者分工在流程的不同阶段，共同构成崩溃安全的完整闭环。
```
