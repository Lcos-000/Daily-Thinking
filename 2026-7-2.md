# 并发安全 Map 选型思考：sync.Map vs 分片锁 vs RWMutex

## 背景

**场景**：调度器需要维护一个 `map[nodeID]*NodeResource`，多个 goroutine 会高频读取（例如调度打分时需要遍历所有节点），而只有少数 goroutine 会低频写入（例如心跳更新节点状态）。

**问题**：在这种“多读少写、且需要全量遍历”的场景下，应该使用哪种并发安全的 Map 结构？

---

## 思考过程

### 第一反应：sync.Map 就很合适？

直觉上，sync.Map 被设计为“适合多读少写”，因此认为直接使用它应该能满足需求，并且它的 `Range` 操作遍历也很快。

**纠正 1：方向反了**

- sync.Map 的“快”是指**针对单个 key 的 Load 操作，当命中 read 层时是无锁的原子读取**，而不是指遍历快。
- `Range` 操作在遇到 `amended=true`（即存在尚未同步到 read 层的新 key）时，需要**加锁将 dirty 层提升并合并到 read 层**，这会退化为全局锁，从而阻塞所有其他操作。
- sync.Map 的官方适用条件之一是“多个 goroutine 读写**不相交**的 key 集合”。而在调度打分场景中，所有 goroutine 都是在读**同一批 key**，这个前提完全不匹配。

### 疑问：能直接读 dirty 层吗？

既然 read 层可能有问题，能不能直接访问 sync.Map 的 dirty 层来读取数据？

**纠正 2：不行**

- `dirty` 是 sync.Map 的私有字段，外部包无法直接访问。
- 即使能访问，dirty 本质上就是一个“加了一把 Mutex 保护的普通 map”，直接读写它等于手写了一个全局大锁方案，性能比分片锁更差。

### 疑问：读 dirty 要加锁吗？

**确认：必须加锁**

dirty 是一个普通的 `map[interface{}]*entry`，不具备任何并发安全能力，完全依赖 sync.Map 内部的 `mu sync.Mutex` 来保护。任何涉及 dirty 的操作（写入新 key、Range 时的提升合并、Load 未命中 read 层后的回退查询等）都必须抢占这把锁。

### 判断转向：分片 map 可能更好

意识到 sync.Map 在遍历场景下的全局锁退化问题后，感觉采用分片 map（每个分片内部各自加锁）会是更好的选择。

**追问：按什么维度分片？**
是按 nodeID 哈希随机分，还是按机架等物理拓扑分？

**回答：两层分片**
第一层按物理节点（如机架）区分，第二层再按哈希做均匀区分，这样既能利用物理拓扑的数据局部性，又能打散热点。

**纠正 3：两层分片增加了读遍历的成本**

- 两层分片解决的是“写热点”问题，但当前场景需要优化的是“全集群遍历打分”这个**读路径**。
- 分片切得越细（例如两层总共分出 40 个分片），一次全量遍历需要获取的锁次数就**越多**。分片数量对“读多次”和“写多次”两种场景的影响是相反的。

### 判断：既然读远多于写，应该少分

在本场景下，读的频次远超写，优化目标应该是“减少读遍历时的拿锁次数”，因此分片数量应该尽可能少（例如 4~8 个），甚至需要反问一句：是否真的有必要分片？

### 反问：难道直接一个 map 就行？

如果分片数量可以少到 4~8 个，那极端一点，岂不是直接用一个 map 就可以了？

**纠正 4：需区分 Mutex 和 RWMutex**

- 如果是一个普通的 `map` 加 `sync.Mutex`，那绝对不行，因为 Mutex 会将所有并发读都压成串行。
- 如果是 `map` 加 `sync.RWMutex`，则是可行的。RWMutex 允许多个读操作并发持有读锁，只在写操作时才阻塞所有读者。这已经解决了分片想要解决的大部分并发读问题。

真正需要评估的问题变成了：**写操作的频率是否高到会让“写阻塞读”成为可观测的性能瓶颈？**

### 收敛结论：一把 RWMutex 或许就是最简方案

在“写是秒级、读是毫秒级”的量级差距下，使用 **`sync.RWMutex` 保护一个普通的 `map[nodeID]*NodeResource`** 就是最简洁且高效的方案。分片锁只有在写并发也非常高的场景下才真正必要。

### 补充疑问：sync.Map 读读之间不是并发的吗？

之前提到 sync.Map 的 dirty 层有全局锁，那是不是意味着它的读和读之间也是无法并发的？

**纠正 5：关键澄清**

- 不准确。sync.Map 的 **read 层读操作之间是完全并发、完全无锁的**（通过原子指针操作实现，甚至比 `RWMutex.RLock` 更轻量）。
- 只有在访问无法命中 read 层、不得不触及 dirty 层时，才会退化为全局互斥锁。

在本调度场景中，心跳写入会不断制造“read 层未同步”的状态，导致大量的 `Range` 或 `Load` 操作大概率触发这条需要加锁的全局路径，因此 sync.Map 反而不划算。

### 追问：为什么 sync.Map 的 dirty 层不用读写锁？

既然 dirty 层的全局互斥锁是性能瓶颈，为什么 Go 官方不把它设计成读写锁，让读 dirty 时也能并发？

**纠正 6：设计哲学**

- sync.Map 的核心目标是**让读路径完全无锁**，而不仅仅是“读写分离”。
- `RWMutex` 虽然允许读并发，但它仍然是锁，涉及原子计数、等待队列等开销。sync.Map 的策略是将 dirty 层设计为**低频兜底路径**，因此选择用最简单的 Mutex 即可，保持整个结构的轻量。
- 如果某个场景导致 dirty 层成为了热点，那只能说明这个场景本来就不适合使用 sync.Map。

### 最终总结：sync.Map 赌的是读无锁

sync.Map 的真正目标是实现读路径的无锁化（仅原子操作），它赌的是“读远多于写、且 key 集合相对稳定”的场景，这样绝大多数读请求都能直接从 read 层原子读取，dirty 层慢一些也无所谓。

而调度器的场景不满足“key 集合稳定”这一关键假设（持续有心跳写入新节点或更新），因此 sync.Map 并不适合。

---

## 方法论沉淀

在进行并发安全 Map 的选型时，应该遵循以下思考路径：

1.  **先看清读写比例和访问模式**：是单 key 操作还是全量遍历？读和写的频次比例如何？
2.  **理解候选方案的核心假设**：
    - `sync.RWMutex + map`：假设写不频繁，允许读并发。
    - 分片锁：假设写并发很高，需要打散写锁争用。
    - `sync.Map`：假设 key 集合稳定，追求读路径完全无锁。
3.  **选择与场景匹配的方案，而非最复杂的方案**：把并发控制的复杂度留给真正需要它的场景。
