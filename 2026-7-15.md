# Operator 的架构拆分与 Reconcile 职责边界

## 问题背景

为自定义资源 `InferenceService` 编写 Operator，需要持续维护实际运行的副本数以匹配期望状态，包括初始部署和根据负载自动扩缩容。具体问题涉及：

- 负载指标该实时查询还是缓存？
- 部分扩容失败时如何处理重试？
- 扩缩容决策是否需要冷却时间以防止震荡？

在进入这些实现细节之前，我们首先对架构层面的设计模式进行了深入探讨。

## 核心架构洞察：两个职责，两个 Controller

### 问题识别

让一个 Reconcile 循环同时承担"扩缩容决策"和"资源创建执行"两类职责，违反了 Kubernetes 的声明式原则与单一职责原则。这两类职责在本质上完全不同：

| 维度 | 扩缩容决策 | 资源创建执行 |
|------|-----------|-------------|
| 变化频率 | 依赖外部动态指标，变化频繁 | 依赖集群资源状态，相对稳定 |
| 数据来源 | 外部 Metrics（Prometheus、资源指标 API 等） | 集群内 Pod/Node 状态 |
| 允许特性 | 可容忍一定滞后 | 需要精确、可靠 |
| 失败处理 | 计算错误可下次修正 | 需要重试、退避、告警 |

### K8s 原生参考：HPA + Deployment

Kubernetes 已有的标准实现正是这种拆分：

- **HPA（决策方）**：持续观测负载指标，计算新的期望副本数，修改 `spec.replicas`
- **Deployment Controller（执行方）**：看到 `spec.replicas` 变化后创建/删除 Pod，不关心"为什么"变化

这两个组件各自是独立的 Controller，有各自的 Reconcile 循环，通过修改资源的 `spec` 字段进行协作，不存在直接调用或共享状态。

### 关键澄清：反对的不是 Reconcile 模式本身

HPA 本身也是 Controller，也有自己的 Reconcile 循环。我们反对的**不是**"用 Reconcile 模式做决策"，而是**一个 Reconcile 循环同时承担两个性质迥异的职责**。拆分为多个单一职责的小 Controller 协作，正是 Kubernetes "多个小 Controller 通过 API 对象协作"这一设计哲学的体现——HPA/Deployment、VPA、Cluster Autoscaler 都遵循同一模式。

## 两个 Controller 的通信协议：spec/status 字段

### 唯一允许的耦合方式

决策 Controller 和执行 Controller 之间的唯一沟通方式：

- **决策 Controller** 修改 `spec.replicas`（表达期望）
- **执行 Controller** Watch `spec.replicas`，据此创建/删除 Pod
- **执行 Controller** 将实际达成的副本数写回 `status.replicas`

不应存在任何其他形式的耦合（直接调用、共享内存、自定义事件通道等）。

## status.replicas 的双重角色

### 角色一：保证决策计算正确性的必要输入

HPA 的核心计算公式本质上是：

```
当前负载总量 ÷ 当前副本数 = 平均负载
平均负载 vs 目标负载 → 新期望副本数
```

**该公式中的"当前副本数"必须使用 `status.replicas`（真实生效值），而非 `spec.replicas`（期望值）**。

**反例**：若 `spec.replicas = 10` 但实际只有 6 个 Pod Running，使用 `spec.replicas` 作为分母会导致：

```
真实负载 / 10 = 被低估的平均负载
```

4 个 Pending 的 Pod 被算入分母但根本没分担负载，HPA 因此错误地认为"负载不高"，可能做出不扩容甚至缩容的错误决策。使用 `status.replicas = 6` 才能反映每个工作 Pod 承受的真实压力。

### 角色二：隐式传递执行层反馈

`status.replicas` 同时充当了执行层向决策层传递反馈的**隐式通道**，无需额外的显式错误上报机制。

**工作方式**：

1. 决策 Controller 将 `spec.replicas` 设为 10
2. 执行 Controller 因资源不足只能创建 6 个，将 `status.replicas` 写为 6
3. 决策 Controller 下次 Reconcile 时读取 `status.replicas = 6`，基于此计算负载
4. 若负载依然偏高（因为只有 6 个 Pod 承担），HPA 会持续尝试上调 `spec.replicas`
5. 直到资源就绪、执行层成功扩容，`status.replicas` 追上 `spec.replicas`，系统收敛

决策 Controller **不需要显式知道"上次扩容失败了"**，但它读的是反映真实情况的 `status`，计算出的下一次决策天然会纳入"没扩起来的那部分"的影响。这是一种通过**状态本身传递反馈**的优雅设计。

## 两个 Controller 的职责边界（最终收敛）

### 执行 Controller（负责 Pod 生命周期）

- **Reconcile 逻辑**：对比 `spec.replicas` 与当前实际 Pod 数量，创建/删除差异部分
- **状态上报**：无论成功与否，将实际达成数量如实写回 `status.replicas`
- **失败处理**：若持续扩容失败，在自身内部做退避重试和告警——这是执行层的职责
- **不做什么**：不将失败信息显式上报给决策 Controller，不做扩缩容决策

### 决策 Controller（HPA）

- **Reconcile 逻辑**：
  1. 读取负载指标
  2. 读取 `status.replicas`（而非 `spec.replicas`）作为当前副本数
  3. 计算新期望副本数，写入 `spec.replicas`
- **不做什么**：
  - 不关心上一次期望是否完全达成
  - 不关心扩容失败的原因
  - 不处理重试、告警等执行层副作用
- **关注点**：仅回答"现在真实情况是什么、目标该调整成什么"

## "HPA 不需要关注扩容失败"的精确含义

这是一个需要精确区分的命题：

- **不关注** = 不处理失败原因、不负责重试策略、不触发告警——这是执行层的职责
- **必需读取** `status.replicas` = 将"真实生效的副本数"作为负载计算的输入——这是决策正确性的基本要求

两者并不矛盾：**HPA 读取 `status.replicas` 是为了"知道真实情况"，而不是为了"处理失败"**。前者是只读输入，后者才是需要产生副作用的控制逻辑。这与单一职责原则完全一致。

## 方法论沉淀

### 1. Controller 拆分原则

当一个 Reconcile 循环需要同时处理以下两类职责时，应拆分为多个独立 Controller：

- **"决定要什么"**：依赖外部动态指标、变化频繁、允许一定滞后
- **"如何实现它"**：依赖集群资源状态、需要精确重试和错误处理

拆分后的 Controller 仅通过资源的 `spec`/`status` 字段进行松耦合协作，这是 Kubernetes 生态中 HPA/Deployment、VPA、Cluster Autoscaler 等成熟设计的共同原则。

### 2. status 字段的设计价值

决策类 Controller 在计算依赖"当前系统规模"的指标时：

- **必须**以反映真实情况的 `status` 字段作为输入，而非表达意图的 `spec` 字段——这是保证决策计算正确性的基本要求
- `status` 字段同时承担起**隐式传递执行层反馈**的角色，使决策层与执行层无需额外的显式错误上报机制即可协同收敛到正确状态
