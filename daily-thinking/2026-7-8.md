# 批量调度接口部分失败的设计思考

## 问题场景

调度器提供批量调度接口，一次提交多个任务，内部逐个尝试调度，可能部分成功、部分失败（如资源不足）。如果接口只简单返回整体的"成功/失败"或一个 HTTP 200，调用方无法知道具体哪些任务成功、哪些失败、失败原因是什么。在不改变"逐个调度、尽力而为"这一处理逻辑的前提下，需要设计一套返回结构和错误语义，让调用方清楚了解这批任务的处理情况，并知道后续该如何应对。

## 核心设计思路

**HTTP 200 + 详细结果列表**，这是业界处理批量部分成功的主流做法（如 Elasticsearch 的 Bulk API）。HTTP 层面返回 200 代表"批量处理"这一动作本身已执行完成，但不代表其中每一项都成功。真正的业务结果通过响应体中的一个列表来详细表达。

## 字段设计演进

### 第一层：仅有成功标识，无法驱动决策

如果结果项只有一个 `success` 布尔字段：

```json
{"task_id": "t8", "success": false}
```

调用方虽能知道谁失败了，却无法知道失败原因。此时对所有失败只能采用一种兜底策略（如整体重试），但这是有问题的：
- "资源不足"类错误值得重试（后续资源可能释放）；
- "参数非法"类错误重试无意义，应直接报错或转人工处理。

因此，必须为调用方提供区分失败原因的途径。

### 第二层：增加失败原因，但程序难以消费

如果增加一个 `reason` 字段，但只是自然语言描述：

```json
{"task_id": "t8", "success": false, "reason": "资源不足，节点无法容纳该任务"}
```

调用方的程序若想据此做自动化判断（如决定是否重试），只能通过字符串匹配 `"资源不足"`，这种方式脆弱且不可靠——描述文本的微小改动（多一个空格、换一种措辞）就会导致逻辑失效。

### 第三层：引入结构化错误码

将失败原因拆分为两部分：

- **错误码**：有限的、结构化的枚举值，供代码做逻辑判断
  - 遇到 `INSUFFICIENT_RESOURCE` → 自动重试
  - 遇到 `INVALID_TASK_PARAM` → 不重试，直接报错
- **错误描述**：人类可读的详细文本，供日志排查、前端展示

这正是接口设计中一个基础但重要的原则：**错误码服务于机器的自动化逻辑判断，描述服务于人的理解与排查**。

## 最终设计方案

每个任务返回三个核心字段：

| 字段 | 说明 |
|------|------|
| `task_id` | 任务标识，用于对应请求中的任务 |
| `status` | `success` 或 `failed`，表示该任务调度是否成功 |
| 成功时 | 返回分配信息（如 `node_id`） |
| 失败时 | 返回 `error_code`（结构化错误码）+ `message`（人类可读描述） |

### 响应示例

```json
{
  "results": [
    {
      "task_id": "t1",
      "status": "success",
      "node_id": "node-a"
    },
    {
      "task_id": "t8",
      "status": "failed",
      "error_code": "INSUFFICIENT_RESOURCE",
      "message": "集群资源不足，无法容纳该任务"
    },
    {
      "task_id": "t9",
      "status": "failed",
      "error_code": "INVALID_TASK_PARAM",
      "message": "任务参数中 CPU 请求量为负数"
    }
  ]
}
```

### 调用方的分类处理逻辑

```
遍历 results：
  ├── status == "success" → 直接使用，记录分配节点
  └── status == "failed"：
        ├── error_code == "INSUFFICIENT_RESOURCE" → 加入重试队列（资源后续可能释放）
        └── error_code == "INVALID_TASK_PARAM" → 不重试，直接告警/转人工处理
```

## 方法论沉淀

设计批量操作类接口时，应遵循两个关键原则：

1. **"整体状态"与"逐项状态"分层表达**
   - 整体用 HTTP 状态码表示批量动作本身的完成情况；
   - 逐项用结果列表详述每一项的业务结果。
   - 不能用单一的成功/失败笼统概括整个批量操作。

2. **逐项失败信息应拆分为机器可判断的结构化错误码与人类可读的描述文本**
   - 错误码驱动自动化的、按错误类型区分的后续处理策略（重试 / 报警 / 人工介入）；
   - 描述文本供问题定位和前端展示；
   - 避免调用方依赖字符串匹配来做决策。

这样才能真正支撑调用方实现健壮、细粒度的错误恢复机制，而不是让所有失败都只能用同一种方式兜底处理。
