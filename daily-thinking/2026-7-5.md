# Day 4：调度请求超时后的幂等性处理

> **场景**：调度器对外提供“提交任务、请求调度”接口。调用方设置 500ms 超时，某次请求中调度器实际已完成挑节点、扣资源、写入分配记录，但响应因网络抖动延迟到第 501 毫秒才发出，调用方已判定超时并断开连接。调用方的自然反应是重试。若调度器没有任何防护，重试会带来什么问题？应该用什么机制正确处理“第一次其实已经成功，只是响应未送达”这种情况？

---

## 一、问题分析

### 1.1 无防护下的重试风险

重试会导致同一任务被调度两次：

- 资源重复扣减
- 分配记录出现重复写入
- 业务语义被破坏（一个任务对应两份调度结果）

### 1.2 初探幂等表思路（存在时序漏洞）

**思路**：调度器本地维护一个状态表，以任务 ID 为 key，写入“已完成”状态，重试时查表拒绝重复处理。

**漏洞**：任务 ID 若由调度器在处理完成后生成，只能通过响应传递给调用方。在响应丢失或延迟的场景下，调用方根本没拿到这个 ID，重试时无法携带，调度器无法将重试请求与原始请求关联起来——重试在调度器看来只是一个全新请求。

---

## 二、关键标识由谁生成

> **核心原则**：去重的关键标识（Idempotency Key）必须由请求发起方在发起请求前就生成好，不能依赖处理方生成后通过响应回传。

- ❌ 调度器生成 ID 并通过响应返回 → 响应超时 = ID 丢失，调用方无法使用
- ✅ 调用方本地生成唯一标识（如哈希值 + 机器号），随请求一起带上

### 2.1 调度器处理逻辑

```
接收请求 → 提取 Idempotency Key
  ├─ 查表：该 Key 已处理？
  │   ├─ 是 → 直接返回此前存储的结果
  │   └─ 否 → 执行调度流程 → 结果与 Key 关联写入状态表 → 返回结果
```

---

## 三、并发场景下的重复调度问题

### 3.1 问题描述

若“原始请求”仍在处理中、尚未写入状态表，而“重试请求”几乎同时到达并查表，会查到“尚无记录”，从而误判为新请求、重复触发调度流程，导致同一任务被调度两次。

### 3.2 乐观锁为什么不适合

乐观锁的核心机制是“读取时记录版本号，写入时校验版本号是否变化”，前提是记录已经存在。此处的冲突场景是两个请求同时试图创建一条原本不存在的记录——状态表中没有记录，也没有版本号可供校验，乐观锁无法直接应用于“防止重复插入”。

---

## 四、正确解法：加锁 + 锁内查表

### 4.1 为什么“先查表、后加锁”是错的

两个并发请求完全可能在加锁之前都查到“无记录”，都误判为应当继续处理，等到各自加锁时才发现被对方抢先，但误判已经发生。

### 4.2 正确顺序：先上锁，后查表

```
1. 请求 A 使用 Key 加锁成功，请求 B（重试）尝试用同一 Key 加锁，因 A 未释放而被阻塞
2. A 在持锁状态下查表，未发现记录 → 执行调度流程 → 写入结果 → 释放锁
3. B 获得锁后查表，此时能查到 A 写入的记录 → 直接返回该结果，不再重复调度
```

查表这一动作始终在“独占访问”状态下进行，杜绝两个请求同时看到“空表”状态的可能。

### 4.3 等效方案：数据库唯一约束

将“检查是否存在”与“写入”合并为数据库层面的一次原子操作（对 Idempotency Key 字段加唯一索引），省去应用层显式加锁。

- 应用层加锁：锁在业务代码中显式控制
- 存储层原子约束：锁交由数据库保证

两者殊途同归。

---

## 五、方法论沉淀

设计幂等机制时：

1. **确定去重标识的生成方与生成时机**——必须由请求方在请求发出前生成，不能依赖响应传递
2. **警惕并发场景下“查表”与“加锁”的时序漏洞**——只有确保“检查状态”和“决定处理与否”这两个动作在独占访问下顺序执行（或通过存储层的原子约束合并这两步），才能真正避免重复处理
```
