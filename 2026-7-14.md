# K8s Scheduler Framework 的 GPU 拓扑感知调度插件

## 原题回顾

为 `kube-scheduler` 编写一个自定义插件，实现 GPU 拓扑感知调度——优先把多 GPU 请求调度到 NVLink 直连的节点。需要实现 `Filter` 和 `Score` 两个扩展点接口，这两个方法会在多个候选节点上被并发调用。

**核心问题：**
- `CycleState`（Filter 和 Score 之间共享数据的上下文对象）在并发场景下是否需要考虑并发安全？
- "节点 GPU 拓扑信息"这份数据该实时查询还是本地缓存？

---

## 第一部分：CycleState 的并发安全

### 1.1 共享粒度：按 Pod 而非按节点

**初始误解：** "每个节点各自一份 CycleState"。

**纠正：** CycleState 实际是按**一次调度周期**（即一个 Pod 的调度过程）分配一份。同一个 Pod 在这次调度决策中，不管候选节点有多少个，所有并发的 Filter/Score 调用共享的是**同一个** CycleState 对象。

- 不同 Pod 的调度周期之间：完全隔离（各自拥有独立的 CycleState）
- 同一个 Pod、多个候选节点之间：共享同一个 CycleState（并发读写）

**框架保障：** CycleState 官方实现的读写接口内部本身是并发安全的（用锁保护），设计者已预见到多个节点的 Filter 会并发写入这个场景。

### 1.2 关键陷阱：框架并发安全 ≠ 你的数据并发安全

Go 的原生 `map` **不是并发安全的**。即便多个 goroutine 写入的是不同的 key，只要发生并发写（或一读一写），Go runtime 会直接 panic（`concurrent map writes`）。

`CycleState.Write(key, val)` 保护的只是"整个 CycleState 内部不同顶层 key 之间"的并发安全，但塞进某个 key 下的 value（如自定义 struct 内部包含一个普通 map），CycleState 并不知道其内部结构，也**不会帮忙保护这个内部 map 的并发写入**。

**错误心态：** "逻辑上不同 key 互不影响，所以并发安全。"

**正确认知：** "逻辑上不同 key 互不影响"与"Go 原生 map 底层实现是否支持并发写"是**两个完全不同层面的问题**。即便逻辑安全，底层数据结构本身的限制依然会导致 panic。

### 1.3 实践方案

```go
type TopologyCache struct {
    mu            sync.RWMutex
    topologyByNode map[string]TopologyInfo
}

func (c *TopologyCache) setNodeTopology(nodeName string, info TopologyInfo) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.topologyByNode[nodeName] = info
}

func (c *TopologyCache) getNodeTopology(nodeName string) (TopologyInfo, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    info, ok := c.topologyByNode[nodeName]
    return info, ok
}
```

**与 Day 1 的关联：** 这与 Day 1 讨论的"节点资源状态该用 sync.Map 还是分片锁/RWMutex"是同一类问题——官方框架解决了"顶层 key 之间"的并发安全，但塞进去的 value 内部的并发保护，仍需自己实现。

### 1.4 锁策略选型：场景决定复杂度

| 维度 | Day 1 场景 | Day 12 场景 |
|------|-----------|------------|
| 数据规模 | 集群规模不确定，可能持续增长 | 候选节点数有明确上限（几十到几百） |
| 对象生命周期 | 长期驻留内存 | 一次调度周期结束后即丢弃 |
| 读写比例 | 读多写少 | 读写接近对半开 |

**结论：**
- 直接排除 `sync.Map`（不符合"读多写少"特征）
- 由于数据规模有明确上限、对象生命周期极短，**一把简单的 `sync.RWMutex` 就是最优解**
- 无需引入分片锁——过度设计在规模小且有明确上限的场景下是负资产

**方法论沉淀：** 机制复杂度要跟问题规模匹配。这与 Day 9 的结论一致。

## 第二部分：GPU 拓扑信息的数据来源

### 2.1 方案对比

| 方案 | 描述 | 评价 |
|------|------|------|
| 方案 A：实时查询 | Filter 每次调用都向节点发起网络请求获取拓扑信息 | ❌ 灾难性——一次调度周期触发几十到几百次网络往返，高并发下会打爆 device plugin |
| 方案 B：本地缓存 | 通过 Informer watch 集群状态到本地内存，Filter/Score 只读本地缓存 | ✅ K8s 真实设计——Scheduler Cache 机制 |

**与 Day 3 的关联：** 方案 A 的并发冲击问题，与 Day 3 讨论的"抢占后多个 goroutine 同时冲击共享资源"是同一类问题的变种。

### 2.2 缓存数据的分类处理策略

**核心洞察：** 不同数据的"变化频率"与"调度决策窗口"的关系，决定了缓存策略。

| 数据类型 | 变化频率 | 读取频率 | 脏读风险 | 缓存策略 |
|----------|----------|----------|----------|----------|
| GPU 拓扑信息 | 极低（硬件级变化，天/月级） | 每次调度 | 趋近于零 | 可完全信赖缓存 |
| 剩余资源量 | 极高（每次分配/释放都变） | 每次调度 | 显著存在（Day 6 的核心问题） | 缓存仅用于粗筛，最终靠原子校验 |

**GPU 拓扑信息为什么可以信赖缓存？**
- 是结构性数据，只有硬件层面变化（物理插拔、固件更新）才会改变
- 变化频率（天级/月级）与读取频率（每次调度）完全不在一个数量级
- 心跳上报机制足以保证新鲜度

### 2.3 关键澄清：兜底重试 ≠ 数据可靠的原因

**错误论证链：** "拓扑信息即使偶尔脏读也没关系，反正最终资源占用有原子锁兜底，失败了重试就行。"

**澄清：** 兜底重试机制解决的是"剩余资源量"这类高频变化数据的脏读问题，而**不是**"拓扑信息几乎不会出错"这件事的原因。两个理由是独立的：

- 拓扑信息可靠 → 靠的是数据变化频率本身极低
- 剩余资源量需要兜底 → 靠的是原子校验 + 重试

混为一谈容易导致"反正有兜底、随便读都不用担心新鲜度"的错误心态，进而让本可避免的脏读大量发生、拖垮系统吞吐。

### 2.4 连接 Day 6：缓存与最终一致性的分层职责

**回退风险识别：** "心跳设置为 5 秒，5 秒内不可能发生特别大变化，就算发生变化导致占用失败，也可以重试。"

**纠正：** 这种表述与 Day 6 已确立的结论存在微妙但关键的偏离：

| 错误心态 | 正确心态 |
|----------|----------|
| "缓存数据大概率是准的，小概率不准，出问题重试兜底" | "缓存数据设计上就该假设它必然是脏的，Filter/Score 只做粗筛，正确性完全不依赖缓存是否新鲜" |
| 可能过度信任缓存做精确打分（如"剩余资源正好够、给最高分"） | 打分逻辑天然留出更宽松的余量，避免在资源边界情况上过度自信 |

**最终对齐的结论：**

> **"缓存能不能用"和"缓存要不要被信任为最终依据"是两回事。**
> - 缓存该用（做粗筛/打分排序，调度器离不开它）
> - 缓存不能被当作"最终能否分配成功"的依据（必须靠原子校验兜底）
> - 这两层职责不能混为一谈

**判断依据：** 数据变化频率与并发读写窗口是否同量级——与"心跳间隔设置得够不够短"无关，也不能用"反正有兜底重试"替代对数据新鲜度本身的重视。

## 方法论沉淀

1. **CycleState 的并发边界：** K8s Scheduler Framework 中，CycleState 是按调度周期（Pod 粒度）共享的对象，框架只保证顶层 key 之间的并发安全，插件自定义的 value 内部数据结构（如 map）仍需自己实现并发保护——这是许多人容易忽视的边界，误以为"逻辑上 key 不冲突"就等于"数据结构层面并发安全"。

2. **并发结构的选型原则：** 是否需要加锁、是否要分片，需要综合读写比例、访问模式，以及**数据规模是否有明确上限、对象生命周期长短**——生命周期极短、规模有界的场景，简单方案往往就是最优解，不必为了"看起来更健壮"而引入不必要的复杂度。

3. **缓存数据的分类处理策略：** 必须按数据本身的变化频率分类，而不能一概而论：
   - 变化频率极低的数据（如硬件拓扑）→ 可以完全信赖缓存
   - 变化频率高、且与并发决策窗口同量级的数据（如剩余资源量）→ 缓存只能用于初步筛选，真正的正确性判断必须放在最终写入/占用的原子校验步骤
   - 判断依据是"数据变化频率与并发读写窗口是否同量级"，不能用"反正有兜底重试"这种心态替代对数据新鲜度本身的重视，否则会导致大量本可避免的脏读拖垮系统吞吐
