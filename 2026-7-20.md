# 多租户推理请求排队的公平性设计

**原题**：推理服务面对多个租户（不同 API Key），所有请求进同一个副本的等待队列，等待被纳入下一轮 Continuous Batching。集群整体负载高、队列积压明显。付费大客户投诉请求经常要等很久，而排在他前面的是大量来自免费用户的、请求量大但单次消耗很小的短请求。调度器该用什么排队策略，既保证付费用户优先被处理，又不会让免费用户完全被饿死？

---

## 第一反应
> 分级队列加加权轮询。

## 追问 1：分级维度的选择
**追问**：认可这是经典解法，但追问具体怎么分级。给出两个候选方案：
- **方案 A**：按用户类型分级（付费/免费各一个队列，不管请求本身大小）。
- **方案 B**：按请求本身的资源消耗分级（重量级/轻量级）。

题目场景中"免费用户请求量大但单次消耗小"暗示：付费用户也可能提交消耗很大的长请求，免费用户也可能有消耗很小的短请求。若只按用户类型分级，加权轮询保证的是"哪个类型的用户优先"，而不是"哪个请求该优先"，这两者是否等同？分级应该按用户身份分，还是按请求资源消耗特征分，还是两者都要？

**回答**：
> 分级按用户等级分，加权按资源消耗加。

## 追问 2：结构性冲突场景
**追问**：认可这是有价值的一步——试图用两个维度分别处理两件事。但追问这两个维度是并列的两层还是应该合并为一个综合分数。

**具体场景**：付费队列（用户等级分级）+ 队列内按资源消耗排序。若付费队列此刻只有一个消耗巨大的长请求（几万 token 输入），免费队列排着 100 个消耗极小的短请求。按"每处理 1 个免费请求处理 3 个付费请求"的加权轮询规则，付费队列没有其他请求时，这个名额是反复处理同一个大请求，还是被浪费/让渡给免费队列？

**进一步指出**："级别"这个维度控制轮询的名额分配比例，"资源消耗"这个维度控制队列内部顺序。这两者是否会互相打架？若付费用户的名额被一个超大请求长期占用（Continuous Batching 下一旦进 batch 就会持续占用资源直到生成完），这一轮的处理效率会不会极低，同时拖累其他本该轮到的免费请求？

**回答**：
> 认为不会，可以把权重分配高一点；如果全都倾向付费用户，免费用户又会被饿死，所以应该靠加权比例来调整。

## 纠正：回避了结构性问题，重新钉死场景
**纠正**：这个回答在回避问题——不是"权重比例该设多少"的问题，而是一个更根本的**结构性问题**：即便权重设置合理，只要"轮询的单位是'一个请求'"，问题就会出现。

**重新钉死场景**：付费:免费 = 3:1 的轮询比例下，某一轮付费队列只有一个消耗巨大（几万 token 输入、且要在 batch 里占用很久）的请求。轮询规则要求处理 3 个付费请求名额，但队列只有这一个，于是这一个请求被塞进去、占用大量资源和时间。

**核心问题**：轮询机制里"一个名额"背后代表的"实际代价"天差地别——可能对应几十毫秒能处理完的短请求，也可能对应占用 batch 资源数分钟甚至更久的长请求。"分级+加权"控制的是"该给哪个队列分配几个名额"，但**没有控制"每个名额背后允许消耗多大资源"**。不管权重调多高，只要队列里混进超大请求，它依然会像"钉子户"一样占用远超"一个名额"应得的实际资源，拖慢同批次的所有其他请求。

**追问**："轮询的单位"应该是"请求个数"，还是应该换成别的东西，才能避免这个问题？提示：这跟 Day 15 讨论过的 Chunked Prefill 解决的问题是否类似？

**回答**：
> 加权配额配的不应该是请求量，而是真实的计算量。

## 确认并落地
**判断**：准确——从"配额按请求个数分配"转向"配额按实际计算量/资源消耗分配"，是 Day 15 Chunked Prefill 思路的直接复用：都是把粗糙的、以个数为单位的配额，换成以真实资源消耗为单位的配额，避免一个大请求打破配额分配的初衷。

**具体落地机制**：
不再说"这一轮该轮到 3 个付费请求"，而是说"这一轮付费用户总共能消耗 **300 个资源单位**（如 token-slot、KV Cache block 数量、算力时间片）的处理配额"。短请求消耗 10 个单位、长请求消耗 200 个单位，配额按消耗量而非请求数分配和轮询。

这样，若付费队列混进一个消耗 250 个单位的超大请求，它会占用这一轮付费配额的大部分（250/300），系统能准确识别"这个大请求已用掉它该占的份额"，不会因为它只是"1 个请求"而误判为"只用了 1/3 的名额"——配额消耗与实际资源占用对齐，不被请求个数这一粗糙单位掩盖真实消耗。

---

## 方法论沉淀
任何涉及"轮询"或"配额分配"的公平性机制，如果分配的基本单位是"个数"（请求数、任务数、连接数等），而这些"个"之间的真实资源消耗差异巨大，那么不管配额比例调得多精细，都无法避免某个大单位打破分配初衷、挤占其他单位应得资源的问题。

**正确做法**：把分配单位从"个数"换成"真实资源消耗量的度量"（token 数、算力时间片、内存占用等），让配额的消耗与实际资源占用直接对齐。

这个原则跨越两种场景，是同一底层哲学的不同应用：
1.  **单个任务内部如何拆分冲击**（如 Chunked Prefill）。
2.  **多个任务之间如何公平分配**（如本题的多租户排队）。
